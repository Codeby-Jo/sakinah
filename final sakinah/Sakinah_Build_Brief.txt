Sakinah — Build Brief
Your scope and the rules. The “how” is yours to figure out.
What you are building
Sakinah · Shukr Mode — the character-first Islamic marriage journey described in your foundation
document and prototype. You build it from the ground up: the screens, the backend, the data, the logic.
This brief gives you the boundary and the rules — not the method.
Where it plugs in
  Frontend: a new features/sakinah/ module inside the existing app you were given. Reuse its auth,
routing, Firebase config, and design system. Do not touch the existing Connections feature.
  Backend: a new Python + FastAPI service that you write. The frontend reaches it via the
VITE_BACKEND_URL already in the env file.
  Data: your own Firestore collections in your throwaway Firebase project.
Non-negotiable rules
  Server-authoritative. The backend verifies the Firebase Auth token and owns every sensitive
write (matches, signals, safety, verification status). Clients must never write match or safety data,
and never decide who is shown. Hiding it in the UI is not enough — it must be impossible from the
client.
  Backend stack is Python + FastAPI. Not negotiable.
  No production access. Work only against your own throwaway Firebase project.
  No real identity data in development. Use the KYC sandbox only (see the KYC brief).
  No AI / Claude / OpenAI keys. Raya is scripted for v1. Do not add paid AI calls.
  Honour the product’s own “must never” list: no swipe, no feed, no public profiles, no photo
before mutual interest, no rejection notifications, no “riya engine” (worship/gratitude/community never
feed matching). Character-first, women-first safety.
